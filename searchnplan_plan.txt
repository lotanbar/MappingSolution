# SearchNPlan — Implementation Plan

---

## Feature Overview

**SearchNPlan** is a single screen that combines two complementary features:

1. **Search** — the user types a query and gets live results from 4 sources (Personal POIs, Imported POIs, OSM, Google Places), ranked by relevance with nearby results first.
2. **Plan** — the user picks multiple destinations from those search results, reorders them, then either launches turn-by-turn navigation or saves the list as a named Plan for later use.

---

## How the User Accesses the Screen

A new **Search button** (magnifying glass icon) is added as the 5th button in the existing bottom action bar on the main map screen (alongside the current Layers, Add POI, Record, and Library buttons). Tapping it navigates to the SearchNPlan screen.

The screen can also be opened from the Library by tapping a previously saved Plan, in which case the destinations are pre-filled.

---

## Complete User Flow (step by step)

### Step 1 — Opening the screen
The user taps the Search button in the bottom nav. The SearchNPlan screen opens as a full-screen sheet over the map. The map remains visible behind the sheet and can be revealed by dragging the sheet down.

### Step 2 — Searching
The user taps the search input at the top and starts typing. Nothing happens until 3 characters have been entered (to avoid spamming APIs on every keystroke). After 3+ characters, results appear automatically after a 300 ms debounce delay. Results update live as the user keeps typing.

### Step 3 — Reading results
Results appear in a flat scrollable list below the input:
- **Personal POIs** the user created that match the query are always shown first (pinned to top). These are pulled instantly from memory — no network needed.
- **Imported POIs** (from GPX imports) that match are shown second, also from memory.
- **OSM results** (Nominatim geocoder) follow, biased toward the user's current location.
- **Google Places results** follow, also biased toward the user's current location.

Each result row shows:
- The POI's icon
- A small colored source badge: "Personal", "Imported", "OSM", or "Google"
- The POI name
- Two **quick action buttons** on the right side of the row:
  - **Navigate** (arrow icon) — immediately launches turn-by-turn navigation to this single destination. No destination list involved.
  - **Add to Plan** (+ icon) — adds this result as a destination in the destination list below. The result list collapses, the destination appears as a card, and the search input re-focuses so the user can search for the next stop.

### Step 4 — Viewing full details (optional)
Tapping the **body** of a result row (not the quick buttons) opens the existing Item Detail screen for that POI. The detail screen shows all available information — name, description, coordinates, photos (for Google Places), media (for personal POIs), etc.

In this context, the detail screen's bottom bar shows two buttons instead of the normal Edit/Delete:
- **Navigate** — same as the quick button, single-stop navigation.
- **Add to Plan** — returns to the SearchNPlan screen with this POI added as a destination.

### Step 5 — Building the destination list
As the user adds destinations (via quick + buttons or from the detail screen), a destination list appears below the search results. Each destination card shows:
- A **drag handle** on the left (three horizontal lines / grip icon) — the user can press and drag to reorder the cards.
- The destination **name**.
- A **remove button** (×) on the right — tapping it removes that destination from the list.

Below the last destination card, there is always a ghost / placeholder row labeled **"Add another destination"**. Tapping it focuses the search input so the user can search for the next stop. This ghost row is always visible as long as the destination list is shown, encouraging the user to keep adding stops.

### Step 6 — Taking action

Once the destination list has at least one item, two buttons appear at the bottom of the screen:

**Navigate button:**
- With 1 destination: tries to open Waze first (if installed), falls back to Google Maps with that single destination.
- With 2+ destinations: always opens Google Maps with all destinations in order as a multi-stop route. (Waze does not support multi-stop natively.)

**Create Plan button:**
- Opens a small name dialog. The name is pre-filled with an auto-generated value like "Plan — 25 Apr 2026". The user can keep it or edit it.
- Tapping Save writes the plan to disk and navigates back to the main map screen. The plan is now accessible in the Library.

### Step 7 — Map preview (Commit 5)
When the user taps a result row, the map (behind the sheet) flies to that location and shows a temporary pin. The user can drag the sheet down to see the map context for the highlighted result, then drag it back up to continue browsing results. Dismissing search clears the temporary pin.

---

## Accessing Saved Plans

In the Library screen, a new **"Plans"** section is added alongside the existing Groups, POIs, and Routes sections. Each plan row shows:
- The plan name
- The number of destinations (e.g., "3 stops")
- The creation date

**Tap** a plan → opens SearchNPlan screen with all destinations pre-filled. The user can navigate, edit the list, and re-save.
**Long-press** → shows a delete option with confirmation.

---

## Who Performs the Location Bias?

**The APIs do it — not the app.** The app just passes the user's current GPS location as a bias coordinate in each API request:
- **Google Places Text Search** (`POST .../places:searchText`): request body includes a `locationBias.circle` centered on the user's location. Google's servers rank results closer to that point higher in the response.
- **Nominatim** (OSM geocoder, `GET .../search?q=...&viewbox=...`): the `viewbox` parameter is set to the current map viewport. Nominatim boosts results within the box but `bounded=0` means global results can still appear.
- **Personal/Imported POIs** (in-memory): the app distance-sorts these itself.

## Data Models

### SearchResult (new sealed class)
```kotlin
sealed class SearchResult {
    abstract val poi: Poi
    data class PersonalPoi(override val poi: Poi) : SearchResult()
    data class ImportedPoi(override val poi: Poi) : SearchResult()
    data class OsmPoi(override val poi: Poi) : SearchResult()
    data class GooglePlace(override val poi: Poi) : SearchResult()
}
```
Wraps the existing Poi model. Source type drives the badge label and the nav route used to open the detail screen.

### Plan (new)
```kotlin
data class Plan(
    val id: String = UUID.randomUUID().toString(),
    val name: String,
    val destinations: List<PlanDestination>,
    val createdAt: Long = System.currentTimeMillis(),
    val updatedAt: Long = System.currentTimeMillis(),
)
```

### PlanDestination (new, same file as Plan)
```kotlin
data class PlanDestination(
    val sourceType: DestinationSource,  // PERSONAL / IMPORTED / OSM / GOOGLE
    val id: String,                      // POI id from its source repository
    val name: String,                    // snapshot — used as fallback if source is unavailable
    val lat: Double,
    val lng: Double,
)
```
Destinations are stored by reference (sourceType + id). When a plan is reopened, the app tries to resolve each destination from its live source. If the POI no longer exists (deleted, or offline), the snapshot name/lat/lng is used instead.

## Detail Screen Integration

The existing ItemDetailScreen already handles all 4 source types (personal POI, bulk/imported POI, OSM POI, Google Place) via the `type` nav argument. The ItemDetailViewModel is unchanged.

The only addition: a `fromSearch: Boolean` nav argument. When true, the bottom bar renders two buttons — Navigate and Add to Plan — instead of the normal Edit/Delete. The "Add to Plan" result is passed back to SearchNPlanScreen via SavedStateHandle (same mechanism as the existing icon-picker flow).

## External Navigation

| Scenario | App | URL / Intent |
|---|---|---|
| 1 destination — Waze installed | Waze | waze://?ll={lat},{lng}&navigate=yes |
| 1 destination — Waze not installed | Google Maps | https://maps.google.com/maps/dir/?api=1&destination={lat},{lng} |
| 2+ destinations | Google Maps (always) | https://maps.google.com/maps/dir/{lat1},{lng1}/{lat2},{lng2}/.../latN,lngN |

Waze does not support multi-stop navigation from a URL scheme. For multiple destinations, Google Maps is always used. Each /lat,lng path segment is one stop in order.

## Technical Notes

- SearchRepository debounces the text input at 300 ms and cancels any in-flight API coroutine on each new keystroke. Personal and Imported POI filtering is synchronous (in-memory) and does not need debouncing.
- OSM search uses Nominatim (text geocoder), not Overpass. Overpass remains unchanged for viewport map display.
- Plans store destinations by reference (sourceType + id) plus a name/lat/lng snapshot. On reopen, the live source is queried first; the snapshot is used if the POI is unavailable (deleted or offline).
- All new screens follow existing patterns: Hilt @HiltViewModel, StateFlow, hiltViewModel() in composables.
- Result order: Personal POIs first (exact prefix match > partial), Imported POIs second (same ranking), then OSM (server-ranked by distance), then Google (server-ranked by distance).
